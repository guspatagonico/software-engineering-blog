Secciones

La envolvente convergente

El modelo mental central del framework
El harness no define un único camino óptimo (frágil) — define una envolvente dinámica que tolera dispersión al inicio y la va cerrando conforme el proceso avanza. El orquestador es el controlador que mantiene esa envolvente activa.

Dos conceptos formales que fundamentan la metáfora

Barrier Functions (CBF)
No solo definís adónde querés llegar — definís regiones prohibidas con "paredes repulsivas". El agente recibe señal de que se está alejando antes de romper algo. En el harness: los MCPs con paths restringidos son barrier functions estructurales, no instruccionales.
Funciones de Lyapunov
Una función que decrece monótonamente a lo largo de cualquier trayectoria del sistema prueba la convergencia. En harness: cada intervención del orquestador debe reducir la distancia al objetivo. Si no la reduce, el feedback loop está roto.

Entropía vs. convergencia

Entropía (harness roto)
Decisiones individualmente razonables que acumulan desviación. El context window se llena de ruido, las herramientas proliferan sin estructura, los sub-agentes optimizan localmente sin coherencia global.
Convergencia (harness bien aplicado)
El espacio de acción está acotado, el feedback loop provee señal clara, las constraints son justas — ni tan laxas que generen entropía, ni tan rígidas que produzcan óptimos locales.

Diseño de sub-agentes

Single responsibility + bounded context
Regla de oro: si describir el scope de un sub-agente requiere más de dos oraciones, está haciendo demasiado.

Anatomía de un sub-agente bien diseñado

Artefacto de salida
Un único tipo: CSS, PHP, SQL, Markdown — nunca mezclados.
Context budget
Lista exacta de archivos que puede ver. Nada más.
Tool budget
MCPs y skills habilitados explícitamente para su scope.
Contrato explícito
Input schema → output schema definidos antes de lanzar.

Anti-patrones

El sub-agente full-stack: acceso a todo el proyecto → entropía garantizada por expansión de scope.

El sub-agente ambiguo: output puede ser CSS o PHP "según lo que haga falta" → el orquestador no puede validar el contrato.

Sin herramientas acotadas: puede leer/escribir cualquier archivo → las barrier functions son instruccionales (débiles) en vez de estructurales (fuertes).

Estrategia de context window

Partición · Handoff · Compresión
El context window es el recurso más escaso. Llenarlo = pérdida de coherencia = entropía. El diseño correcto no es usar ventanas más grandes — es que cada sub-agente opere con el contexto mínimo suficiente.

Context Budget (template)

## Context Budget: [nombre-subagente]

Sistema:    ~800 tokens   (AGENTS.md específico del sub-agente)
Spec task: ~1500 tokens   (solo la sección relevante del spec)
Archivos:  ~3000 tokens   (solo los archivos que necesita tocar)
Handoff:    ~500 tokens   (resumen del estado de la fase anterior)
─────────────────────────────────────
Total:     ~5800 tokens   (margen amplio para el output)

NO entra: historial previo, archivos de otros módulos,
la spec completa del proyecto.

Handoff comprimido (template)

## Handoff: [nombre-subagente] → [destino]
Completado:   [qué hizo, en una oración]
Artefactos:   [lista de archivos con paths exactos]
Estado clave: [una o dos líneas relevantes para el siguiente agente]
Pendiente:    [qué queda fuera de scope]
Blockers:     ninguno | [descripción si hay alguno]
Resultado:    DONE | PARTIAL | BLOCKED

Ejemplo real — pivot.lat

## Handoff: pmpro-css → pmpro-js
Completado:   Estilos del pricing table con SVG column icons y badge "Mejor Valor"
Artefactos:   /wp-content/themes/pivot/css/pmpro-table.css
Estado clave: La clase .pmpro-best-value marca la columna destacada;
               JS debe toggle-arla via data-attribute, no inline styles
Pendiente:    Responsive breakpoints en mobile (fuera de scope)
Blockers:     ninguno
Resultado:    DONE

Session-state.md — fuente de verdad global

El orquestador mantiene un archivo comprimido que es el único estado global del proyecto. Entra en el context del orquestador en cada turno, pero nunca en el de los sub-agentes.

Orquestación paralela

Fork-Join · Detección de drift
El orquestador no piensa sobre el dominio — mide distancia al objetivo y redirige. Su función es mantener la envolvente convergente activa.

Patrón Fork-Join

┌─ Sub-agente A (CSS/frontend) ────┐
Orquestador ───────►├─ Sub-agente B (PHP/backend) ─────┤──► Join ──► Validación ──► siguiente
└─ Sub-agente C (DB/queries) ─────┘

Condiciones para paralelizar (las tres deben cumplirse)

1. Los sub-agentes no escriben al mismo archivo
2. Sus outputs son independientes en esta fase (no se alimentan entre sí)
3. El orquestador puede validar cada output aisladamente

Señales de drift

SeñalInterpretaciónAcción del orquestador
Output más extenso de lo esperadoExpansión de scopeInterrumpir, re-definir scope
Modificaciones a archivos fuera del scopeViolación de barrier functionRevertir, separar en nuevo sub-agente
Preguntas sobre dominio ajeno al taskAmbigüedad en el specAclarar y proveer contexto mínimo
Output que no matchea el schema de handoffContrato rotoRe-ejecutar con spec más restrictiva
Loops de auto-corrección (>2 iteraciones)Convergencia prematuraCambiar enfoque o partir el task

Skills & MCPs como constraints

Barrier functions estructurales
El insight clave: skills y MCPs no son solo herramientas de capacidad — son mecanismos de constraint estructurales. Asignar a un sub-agente solo los MCPs relevantes define las paredes de su envolvente de forma física, no instruccional.

Constraint instruccional vs. estructural

Instruccional (débil)
"No toques los archivos PHP." El agente puede ignorarla, especialmente bajo drift o cuando el contexto es ambiguo.
Estructural (barrier function real)
El MCP de filesystem solo tiene acceso a /css/. Físicamente no puede tocar el PHP. El costo de violar el scope es infinito.

Checklist de harness design

Validar antes de lanzar cada proceso agéntico
0 / 15 completados

Por cada sub-agente

✓
¿Tiene un único tipo de artefacto de salida definido?
✓
¿Su context budget está definido (lista exacta de archivos que puede ver)?
✓
¿Su tool budget está definido (MCPs y skills habilitados explícitamente)?
✓
¿Su contrato con el orquestador es explícito (input schema → output schema)?
✓
¿Su AGENTS.md o system prompt es específico al scope (no genérico)?

Para el orquestador

✓
¿Existe un session-state.md como única fuente de verdad del estado global?
✓
¿El formato de handoff comprimido está definido para cada sub-agente?
✓
¿Están identificadas las fases paralelizables vs. secuenciales?
✓
¿Están definidas las señales de drift para cada sub-agente?
✓
¿Hay un criterio claro de convergencia (cómo sabe el orquestador que un output es válido)?

Context window

✓
¿Cada sub-agente recibe solo los archivos que necesita?
✓
¿El historial de fases anteriores se poda antes de cada sub-agente?
✓
¿Los handoffs son resúmenes estructurados, no outputs completos?
✓
¿La spec del proyecto se fragmenta y solo entra la sección relevante?
✓
¿El session-state.md está actualizado antes de lanzar la siguiente fase?

Vocabulario del framework

Términos operacionales
envolvente convergente
El rango de comportamiento tolerable del agente, que se estrecha progresivamente hacia la solución. La metáfora central del framework.
entropía agéntica
Acumulación de drift: decisiones individualmente razonables que producen un resultado incoherente. Señal de harness roto.
convergencia
El proceso llega a la solución óptima dentro de las cotas definidas. Objetivo de todo proceso agéntico bien harnesseado.
barrier function
Constraint estructural (no instruccional) que hace imposible — no solo costosa — la violación del scope. Ej: MCP con paths restringidos.
context budget
Presupuesto de tokens asignado a un sub-agente: qué puede ver y cuánto. Se define antes de lanzar, no durante.
tool budget
Conjunto de MCPs y skills habilitados para un sub-agente específico. Define sus barrier functions estructurales.
handoff comprimido
Resumen estructurado del output de un sub-agente, optimizado para entrar barato en el context del siguiente. No es el output completo.
session state
Estado global comprimido del proyecto, mantenido exclusivamente por el orquestador. Nunca entra en el context de los sub-agentes.
drift
Desviación acumulada del objetivo. Señal de que el agente se acerca a los límites de la envolvente y requiere intervención del orquestador.
fork-join
Patrón de ejecución paralela: múltiples sub-agentes independientes seguidos de un punto de validación conjunta por el orquestador.
contrato de sub-agente
Definición explícita del input schema y output schema de un sub-agente. El orquestador valida contra este contrato.
