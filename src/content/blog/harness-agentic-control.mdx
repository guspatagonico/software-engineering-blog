    <h2>Tesis y alcance</h2>
    <p>
      Un agente con herramientas es un sistema de control: el harness define el conjunto de
      estados permitidos, cómo se observan y qué señales lo devuelven a la trayectoria correcta.
      La tesis central es que la calidad del resultado no depende solo del prompt, sino del
      circuito de observabilidad + corrección que rodea al modelo: contratos explícitos, límites
      operativos y métricas verificables que permitan detectar deriva temprana.
    </p>
    <p>
      Cuando un agente usa tools, se vuelve un proceso iterativo: propone, ejecuta, valida y
      vuelve a proponer. Ese loop necesita fricciones deliberadas (presupuestos de contexto,
      listas de archivos permitidos, validadores de salida) para evitar que una buena intuición
      termine en una implementación incoherente. En términos prácticos, el harness hace que el
      progreso sea auditable y reproducible, no solo persuasivo.
    </p>
    <p>
      En esta nota propongo un modelo operativo para OpenCode y agentes similares: contratos por
      fase (inputs/outputs), budgets de herramientas, y señales de deriva medibles. La intención
      no es “hacer al modelo más inteligente”, sino mejorar su estabilidad: reducir entropía,
      acotar el espacio de acción y acelerar la convergencia hacia resultados verificables.
    </p>
    <p>Idea clave: sin telemetría y criterios de aceptación no hay control, solo narrativa.</p>
    <ul>
      <li>No cubre entrenamiento, fine-tuning ni alineación a nivel de modelo base.</li>
      <li>No es una guía de seguridad completa; asume tools confiables y políticas definidas.</li>
      <li>Se centra en entornos de ingeniería (repos, artefactos, validación) y no en UX final.</li>
      <li>No reemplaza revisión humana en decisiones críticas; la automatización es asistida.</li>
    </ul>
    <Highlight>
      La tesis: el rendimiento de un agente no mejora solo con mejores prompts, sino con un
      harness que haga verificable el progreso, reduzca entropía y permita convergencia.
    </Highlight>
  </div>

<div class="panel" id="panel-definitions">
  <h2>Definiciones clave y contexto</h2>
  <div class="card-stack">
    <Card title="Harness">
      Conjunto de restricciones, contratos y observabilidad que gobierna cómo un agente puede usar
      herramientas, qué puede ver y cómo reporta su avance. No es solo “reglas”, sino un sistema
      operativo: delimita el espacio de acción, impone validaciones y habilita señales de control
      para medir progreso real.
    </Card>
    <Card title="Contrato">
      Especificación de inputs y outputs por fase con criterios de aceptación. Funciona como API
      interna entre orquestador y sub-agentes y evita ambigüedad: define qué evidencias son válidas,
      qué formato debe tener la salida y cómo se verifica.
    </Card>
    <Card title="Deriva">
      Desalineación entre el objetivo del sistema y el estado actual del agente. Se detecta por
      señales objetivas (p. ej. cambios fuera de scope, tool calls no autorizadas, métricas que no
      mejoran) y se corrige con límites o re-planificación. La clave es medir el desvío, no
      asumirlo.
    </Card>
    <Card title="Observabilidad">
      Instrumentación mínima para que el loop sea verificable: logs de tools, checklist de
      artefactos, hashes de archivos y métricas de convergencia. Sin telemetría, el orquestador
      decide a ciegas.
    </Card>
    <Card title="Presupuesto (budget)">
      Límite operativo de contexto, tools o pasos por fase. Obliga a priorizar y evita loops
      infinitos. También sirve como señal: si el agente consume el budget sin progresar, hay deriva
      o un contrato mal definido.
    </Card>
  </div>
</div>

  <div class="panel" id="panel-model">
    <h2>Modelo del sistema</h2>
    <p>
      <strong>Idea base:</strong> en sistemas agénticos con tool use (uso de herramientas externas por
      el modelo: APIs, CLI, archivos), el harness es el plano de control. Decide qué ve el agente,
      qué puede ejecutar y cómo se valida cada paso.
    </p>
    <p>
      <strong>Por qué importa:</strong> este modelo aplica a muchas arquitecturas (APIs con tool calling,
      orquestadores con validadores y flujos iterativos) porque crea un loop verificable y evita que
      el agente avance solo por “parecer correcto”.
    </p>
    <ul>
      <li><strong>Intención:</strong> objetivo y contrato que definen lo que significa “correcto”.</li>
      <li><strong>Ejecución:</strong> tool calls y escritura de artefactos dentro del scope permitido.</li>
      <li><strong>Validación:</strong> checks y criterios de convergencia para aceptar o corregir.</li>
    </ul>
    <p>
      <strong>En OpenCode:</strong> el patrón se materializa con contratos por fase, budgets de herramientas,
      artefactos verificables y checkpoints que el orquestador usa para decidir si continuar o
      corregir.
    </p>

    <h3>Loop de control</h3>
    ```mermaid
    flowchart TD
      A[Input + Contrato] --> B[Agente]
      B --> C["Llamadas a tools"]
      C --> D[Artefactos]
      D --> E[Validador]
      E --> F{"¿Convergió?"}
      F -- "Sí" --> G[Publicar]
      F -- "No" --> H["Delta + Nuevas restricciones"]
      H --> B
    ```
    <p>
      Este loop es el corazón de Harness Engineering: cada vuelta termina con un veredicto.
      “Convergió” no significa “suena bien”, sino “cumple contrato”: outputs con formato válido,
      archivos dentro del scope y métricas aceptables. El nodo “Delta + Nuevas restricciones” es
      donde el orquestador ajusta budgets o recorta contexto para reducir deriva.
    </p>

    <h3>Loop de tool use (agéntico)</h3>
    ```mermaid
    sequenceDiagram
      participant O as Orquestador
      participant A as Agente
      participant T as Herramientas
      O->>A: Tarea + Contrato + Presupuestos
      A->>T: tool_use (call)
      T-->>A: tool_result
      A->>O: Salida + Evidencia
      O->>A: Aceptar / Revisar
    ```
    <p>
      En cualquier stack con tool calling, el contrato debe viajar con la tarea: el agente no solo
      devuelve una respuesta, devuelve evidencia (diffs, logs, checks) que el orquestador valida.
      Esa distinción hace que el control sea objetivo: se acepta o se pide revisión en función de
      datos, no de persuasión. OpenCode materializa este patrón con artefactos y criterios de
      aceptación verificables.
    </p>

    <h3>Arquitectura (contenedores)</h3>
    ```mermaid
    flowchart LR
      user[Usuario]
      orchestrator[Orquestador\nLoop de control]
      agent[Agente\nLLM + politicas]
      tools[Herramientas\nMCP/skills]
      artifacts[Artefactos\nArchivos, logs, checklists]

      user -->|Define objetivo| orchestrator
      orchestrator -->|Tarea + Contrato| agent
      agent -->|Uso de herramientas| tools
      agent -->|Escribe| artifacts
      orchestrator -->|Valida| artifacts
    ```
    <p>
      Esta vista resume los límites del sistema. El orquestador es el único actor que valida y
      decide avance; el agente ejecuta dentro de restricciones; las herramientas son capacidades
      explícitas; y los artefactos son la fuente de verdad. Este reparto de responsabilidades es
      común en frameworks de agentes con herramientas; en OpenCode, toda decisión crítica vuelve a
      los artefactos, no al chat.
    </p>

<div class="panel" id="panel-hypotheses">
  <h2>Hipótesis y conjeturas</h2>
  <div class="post-grid">
    <Card title="H1: Budgeted tools reduce drift">
      Si el harness limita herramientas por fase (tool budget), la deriva disminuye. Se falsifica si
      la tasa de rework o el número de tool calls innecesarias no baja.
    </Card>
    <Card title="H2: Contratos mejoran la convergencia">
      Contratos con outputs verificables (schemas o checklists) reducen ciclos. Se falsifica si el
      tiempo a convergencia no mejora comparado con prompts libres.
    </Card>
    <Card title="H3: Señales tempranas evitan cascadas">
      Detectar cambios fuera de scope o saltos de archivos reduce errores en etapas posteriores. Se
      falsifica si los defectos de fase tardía no cambian.
    </Card>
  </div>
</div>

<div class="panel" id="panel-tradeoffs">
  <h2>Tradeoffs y modos de falla</h2>
  <ul>
    <li>
      <strong>Rigidez vs. creatividad:</strong> demasiadas restricciones reducen exploración; muy
      pocas producen entropía.
    </li>
    <li>
      <strong>Observabilidad vs. costo:</strong> más logs y checkpoints dan trazabilidad pero
      aumentan latencia y tokens.
    </li>
    <li>
      <strong>Herramientas amplias vs. seguras:</strong> tools potentes aceleran, pero amplían la
      superficie de error y seguridad.
    </li>
    <li>
      <strong>Fallo típico:</strong> orquestador sin criterio de convergencia claro =&gt; loop
      infinito.
    </li>
  </ul>
</div>

<div class="panel" id="panel-patterns">
  <h2>Patrones de implementación</h2>
  <div class="card-stack">
    <Card title="Phase contract">
      Cada fase define: inputs permitidos, archivos visibles, tools habilitados, y output schema.
    </Card>
    <Card title="Drift sentinels">
      Checks automáticos: cambios fuera de ruta, diff size &gt; threshold, tool calls no aprobadas.
    </Card>
    <Card title="Evidence bundling">
      Cada output incluye evidencias: diff, logs, y rationale. El orquestador valida antes de pasar
      de fase.
    </Card>
  </div>
</div>

<div class="panel" id="panel-evaluation">
  <h2>Evaluación y medición</h2>
  <ul>
    <li>
      <strong>Time-to-converge:</strong> iteraciones hasta pasar validación.
    </li>
    <li>
      <strong>Drift rate:</strong> porcentaje de steps con violaciones de contrato.
    </li>
    <li>
      <strong>Tool efficiency:</strong> tool calls útiles / tool calls totales.
    </li>
    <li>
      <strong>Rework ratio:</strong> líneas revertidas vs líneas aceptadas.
    </li>
  </ul>
  <Highlight> Medir es parte del harness: sin métricas, no hay feedback loop real. </Highlight>
</div>

  <h2>Direcciones futuras</h2>

    <li>Tool search dinámico para reducir tokens de definiciones.</li>
    <li>Contratos ejecutables con schemas y validadores automáticos.</li>
    <li>Benchmarks de harness: medir convergencia, drift y seguridad en entornos reales.</li>
  </ul>

  <div class="panel" id="panel-summary">
    <h2>Resumen y takeaways</h2>
    <ul>
